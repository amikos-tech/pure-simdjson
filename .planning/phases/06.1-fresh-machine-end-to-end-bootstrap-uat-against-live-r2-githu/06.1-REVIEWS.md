# Cross-AI Plan Review - Phase 06.1 (Fresh-machine end-to-end bootstrap UAT against live R2 + GitHub Releases)

## Gemini Review

### Gemini Output

#### Summary

The plans address the Phase 06.1 goal well: validate the live public bootstrap path on clean runners through a dedicated rerunnable workflow, verify artifacts against published metadata, and keep post-publish validation decoupled from `release.yml`. Gemini called out the dual-checkout pattern and the forced source-ladder split between R2 and GitHub fallback as the core strengths.

#### Strengths

- Dual-checkout architecture keeps workflow tooling on the current branch while validating the published tag in `target-src/`.
- Explicit source routing via `PURE_SIMDJSON_DISABLE_GH_FALLBACK=1` and a deliberately broken `PURE_SIMDJSON_BINARY_MIRROR` prevents one source from masking the other.
- Pass criteria are stronger than "parse succeeded": the plans require empty-cache start, expected cache path creation, and unix `0700` permission evidence.
- Local contract tests reduce silent workflow drift.

#### Concerns

- [MEDIUM] Cross-platform permission checks are easy to get wrong because GNU `stat` and BSD `stat` use different flags; a naive implementation could break macOS validation.
- [LOW] The wrapper deletes the requested cache dir, so it should guard against empty or dangerous `--cache-dir` inputs.

#### Suggestions

- Use a portable permission check strategy instead of assuming one `stat` flavor.
- Add a safety guard before recursive cache-dir deletion.
- Format `$GITHUB_STEP_SUMMARY` output clearly so matrix job summaries stay readable.

#### Risk Assessment

LOW. The architecture is solid and the likely issues are localized shell-script details rather than plan-level gaps.

## the agent Review

### the agent Output

#### Summary

The three-plan sequence is well-shaped: a reusable smoke wrapper first, then the dedicated `workflow_dispatch` validation workflow plus contract tests, then the operator-facing documentation. The design stays within the locked phase boundary, validates both the normal R2 path and the forced GitHub fallback path, and avoids coupling live validation into the publish workflow.

#### Strengths

- Clean dependency chain: `06.1-01 -> 06.1-02 -> 06.1-03`.
- Dual-checkout against `target-src/` at `refs/tags/v${version}` is the load-bearing architectural decision.
- Explicit env-based source forcing makes the wrong download ladder impossible rather than merely unlikely.
- Verification is cheap and grep-testable.
- The plans reuse the existing bootstrap smoke and established workflow contract-test pattern.

#### Concerns

- [MEDIUM] Plan 01 needs an explicit portable permission-check approach; Linux and macOS `stat` syntax differ.
- [MEDIUM] Windows workflow legs need guaranteed Bash execution for the wrapper script.
- [MEDIUM] The plan should make checkout ordering and the `target-src/internal/bootstrap/version.go` validation explicitly fail-fast before smoke runs.
- [MEDIUM] The forced broken-mirror assumption should be validated carefully; if the CDN does not return a hard 404 for both metadata and artifact paths, the fallback proof can become ambiguous.
- [LOW] Contract tests are useful but somewhat brittle because they rely on many text matches.
- [LOW] Job-level timeouts are missing.

#### Suggestions

- Set `defaults.run.shell: bash` at workflow or job level.
- Detect platform before reading unix permission bits, or choose a portable alternative.
- Make the `target-src` checkout path and version check explicit in acceptance criteria.
- Add `timeout-minutes` to matrix jobs.
- Document the fallback subset tradeoff more explicitly if `linux-arm64` is intentionally excluded from the fallback matrix.

#### Risk Assessment

MEDIUM-LOW. The architecture is correct; the main execution risk is runner portability and CI hardening, not scope or dependency structure.

## Consensus Summary

Both reviewers think the plan architecture is sound and appropriately scoped. The main risks are not conceptual; they are in cross-platform shell behavior and runner hardening details that can turn an otherwise good design into a flaky implementation.

### Agreed Strengths

- The three-plan split is clean and the dependency order is sensible.
- The dual-checkout strategy is the right way to validate published-tag behavior without depending on old workflow definitions.
- The R2-only versus forced-GitHub-fallback split gives credible, non-overlapping coverage of both source ladders.
- The phase requires stronger evidence than a passing parse alone: empty cache, exact cache-path creation, and unix permission assertions.
- Reusing the existing smoke harness and adding local contract tests keeps the implementation grounded in existing repo patterns.

### Agreed Concerns

- Cross-platform shell portability needs to be nailed down before execution, especially permission checks on Linux versus macOS and Bash execution on Windows runners.
- The wrapper/workflow should include stronger runner-side guardrails so CI behavior is deterministic and safe: explicit shell selection, fail-fast checkout/version validation, safe cache cleanup, and bounded job runtime.

### Divergent Views

- Gemini rates the plans as LOW risk and sees the likely issues as minor shell details; the agent rates them MEDIUM-LOW risk because it also worries about Windows shell defaults, workflow ordering, and timeout hardening.
- Gemini's unique operational concern is unsafe cache-dir deletion; the agent's unique concerns are workflow shell defaults, missing job timeouts, and the assumption that the broken mirror always yields a clean fallback signal.
