## User Constraints (from CONTEXT.md and current codebase)

### Locked Decisions

- **D-01:** Phase `06.1` runs as a CI-only fresh-runner matrix across all five supported targets.
- **D-02:** For this phase, "fresh machine" means a clean hosted runner with an empty `pure-simdjson` cache, not a manually prepared workstation.
- **D-03:** The normal validation path uses live R2 artifacts on all five supported targets.
- **D-04:** GitHub Releases fallback must also be validated, but only on a smaller representative subset instead of the full five-target matrix.
- **D-05:** Every validation run must begin from an empty cache and then exercise the real bootstrap path through `NewParser() -> Parse([]byte("42")) -> Root().GetInt64()`.
- **D-06:** A pass requires asserting that the expected platform cache file was created in the bootstrap cache layout after download.
- **D-07:** Unix targets must also assert that the created cache directory keeps the locked `0700` permission contract from Phase 5.
- **D-08:** Phase `06.1` ships as a separate rerunnable workflow keyed by a published tag or version, not as an inline extension of `release.yml`.
- **D-09:** The public-validation workflow must stay outside the publish workflow so operators can rerun it against an already-published release without rebuilding or republishing artifacts.
- **D-10:** Manual scripts and runbook steps may support the implementation, but the canonical operator entrypoint for this phase is the dedicated rerunnable workflow.

### Current Codebase Correction

- The original Phase 5 human-UAT wording assumed `internal/bootstrap/checksums.go` would be populated at release time.
- The current code no longer works that way in production:
  - `internal/bootstrap/checksum_resolution.go` resolves expected digests from published `SHA256SUMS` metadata under the requested tag.
  - `internal/bootstrap/checksums.go` is now only an optional override/test seam.
- Phase `06.1` therefore must validate the live published `SHA256SUMS` path, not a CI-populated in-source checksum map.

### the agent's Discretion

- Exact workflow filename, as long as it is a dedicated rerunnable workflow under `.github/workflows/`.
- Exact representative GitHub fallback subset, as long as it covers each OS family once and stays smaller than the five-target R2 matrix.
- Exact script/test split between shell and Python, as long as the workflow contract stays grep-testable and the live smoke path is explicit in job logs.

### Deferred Ideas (OUT OF SCOPE)

- Corporate-firewall or proxy-specific validation remains separate backlog work.
- Re-merging public validation into `release.yml`.
- Any validation path that relies on local `bootstrap.Checksums` overrides instead of the published `SHA256SUMS` artifacts.

## Summary

Phase `06.1` should ship as a **dedicated post-publish validation workflow** plus a **reusable live-smoke wrapper script**:

1. **Workflow:** add `.github/workflows/public-bootstrap-validation.yml` with `workflow_dispatch` and required `version` input like `0.1.0`.
2. **Target source checkout:** keep the workflow itself on the current branch, but check out the requested tag into `target-src/` so `internal/bootstrap/version.go` and `tests/smoke/go_bootstrap_smoke.go` come from the published source being validated.
3. **R2 full matrix:** validate `linux/amd64`, `linux/arm64`, `darwin/amd64`, `darwin/arm64`, `windows/amd64` with `PURE_SIMDJSON_DISABLE_GH_FALLBACK=1`.
4. **GitHub fallback subset:** validate `linux/amd64`, `darwin/arm64`, and `windows/amd64` by forcing the mirror base to a guaranteed-404 path and leaving fallback enabled.
5. **Wrapper script:** add `scripts/release/run_public_bootstrap_smoke.sh` that clears the cache dir, runs `go run ./tests/smoke/go_bootstrap_smoke.go` from `target-src`, asserts the expected cached artifact exists, asserts unix `0700` dirs, and writes an explicit summary when `GITHUB_STEP_SUMMARY` is available.

This model satisfies all locked decisions while preserving the current architecture:

- it validates the real public ladder against live bytes and live `SHA256SUMS`,
- it avoids modifying `release.yml`,
- it stays rerunnable for any already-published tag,
- it does not depend on the target tag containing the new workflow/scripts because the workflow checks out the target tag into a subdirectory.

## Recommended Workflow Model

### 1. Dedicated workflow

**Recommended file:** `.github/workflows/public-bootstrap-validation.yml`

**Required behavior:**

- `workflow_dispatch` with required `version` input, semver without leading `v`
- `concurrency: public-bootstrap-validation-${{ inputs.version }}`
- permissions limited to `contents: read` and `actions: read`
- pinned `actions/checkout` and `actions/setup-go` SHAs matching the release workflow
- two separate jobs:
  - `validate-r2` — five-target matrix, fallback disabled
  - `validate-gh-fallback` — representative three-target subset, fallback forced

### 2. Dual checkout pattern

The workflow must not rely on the current branch's `Version` constant when validating an older published tag.

Use:

- checkout #1: current branch at repo root for workflow-owned scripts and tests
- checkout #2: `refs/tags/v${{ inputs.version }}` into `target-src/`

Then validate:

- `target-src/internal/bootstrap/version.go` matches `${{ inputs.version }}`
- `target-src/tests/smoke/go_bootstrap_smoke.go` exists

This keeps the workflow rerunnable after the repository advances beyond the validated tag.

### 3. Representative GitHub fallback subset

Recommended subset:

- `linux/amd64`
- `darwin/arm64`
- `windows/amd64`

Rationale:

- linux covers the primary shared-object path and unix permission assertions,
- darwin covers the macOS `.dylib` family without duplicating both darwin arches,
- windows covers the DLL naming / loader path that diverges from unix.

Validating both linux arches or both darwin arches in the fallback subset would increase cost without adding new source-ladder signal.

### 4. Force the source ladder explicitly

**R2 job behavior**

- leave `PURE_SIMDJSON_BINARY_MIRROR` unset
- export `PURE_SIMDJSON_DISABLE_GH_FALLBACK=1`
- clear `PURE_SIMDJSON_CACHE_DIR`

If the smoke passes, it proves the live public R2 path works because GitHub fallback is impossible in that mode.

**GitHub fallback job behavior**

- export `PURE_SIMDJSON_BINARY_MIRROR=https://releases.amikos.tech/pure-simdjson-does-not-exist`
- leave `PURE_SIMDJSON_DISABLE_GH_FALLBACK` unset
- clear `PURE_SIMDJSON_CACHE_DIR`

Because both the metadata and artifact URLs under that broken R2 base 404, success proves the library used the GitHub Releases fallback for both `SHA256SUMS` and the artifact bytes.

## Implementation Guidance

### `scripts/release/run_public_bootstrap_smoke.sh`

The wrapper should accept:

- `--repo-root <path-to-target-src>`
- `--version <semver-without-v>`
- `--goos <linux|darwin|windows>`
- `--goarch <amd64|arm64>`
- `--mode <r2|github-fallback>`
- `--cache-dir <path>`

It should:

1. remove the cache dir before running
2. compute the expected artifact filename exactly as `PlatformLibraryName(goos)` does:
   - linux -> `libpure_simdjson.so`
   - darwin -> `libpure_simdjson.dylib`
   - windows -> `pure_simdjson-msvc.dll`
3. run `go run ./tests/smoke/go_bootstrap_smoke.go` from `--repo-root`
4. assert the cached file exists at:
   - `<cache-dir>/v<version>/<goos>-<goarch>/<libname>`
5. on unix, assert `0700` on:
   - `<cache-dir>`
   - `<cache-dir>/v<version>`
   - `<cache-dir>/v<version>/<goos>-<goarch>`
6. append summary lines to `GITHUB_STEP_SUMMARY` so workflow runs show:
   - validation mode
   - platform
   - expected cache path
   - whether GitHub fallback was enabled
   - observed unix perms when relevant

### Workflow contract tests

Add `scripts/release/test_public_bootstrap_validation_contracts.py` to protect against drift.

It should assert:

- the workflow is `workflow_dispatch`
- `version` input is required
- concurrency uses `public-bootstrap-validation-${{ inputs.version }}`
- both `actions/checkout` and `actions/setup-go` are pinned by full SHA
- the workflow checks out `target-src`
- the R2 job includes all five targets
- the fallback job includes `linux-amd64`, `darwin-arm64`, and `windows-amd64`
- both jobs call `scripts/release/run_public_bootstrap_smoke.sh`
- the script still contains:
  - `PURE_SIMDJSON_DISABLE_GH_FALLBACK=1`
  - `PURE_SIMDJSON_BINARY_MIRROR=https://releases.amikos.tech/pure-simdjson-does-not-exist`
  - the expected cache-path formula

### Documentation boundary

`docs/releases.md` should add a **post-publish public bootstrap validation** section that points operators at the new workflow after `release.yml` succeeds.

`docs/bootstrap.md` should keep the existing separation:

- local/bootstrap-test coverage uses `httptest` plus override seams
- live public CDN / GitHub Releases validation belongs to Phase `06.1`

### Failure Gates

| Fault | Test Pattern | Expected Behavior |
|-------|--------------|-------------------|
| workflow validates current branch code instead of the requested tag | explicit `target-src` checkout + version guard | job fails before smoke if `target-src/internal/bootstrap/version.go` drifts from input |
| stale cache makes bootstrap look green without download | wrapper deletes cache dir before smoke | job fails if the expected downloaded artifact is not recreated |
| R2 job silently falls back to GitHub | wrapper exports `PURE_SIMDJSON_DISABLE_GH_FALLBACK=1` | R2 job fails instead of masking a public-R2 outage |
| fallback job still downloads from R2 | wrapper forces broken mirror base | success implies GitHub Releases fallback was used |
| unix perms regress but parse still works | wrapper stats cache dirs after smoke | unix jobs fail on non-`0700` cache dirs |
