# Phase 06.1: Fresh-machine end-to-end bootstrap UAT against live R2 + GitHub Releases - Context

**Gathered:** 2026-04-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Validate the already-published bootstrap flow against live public artifacts on the five supported targets from clean, empty-cache runners. Phase `06.1` proves that `NewParser()` can download a real artifact, verify it against release-populated checksum metadata, cache it in the expected location, and parse a sample document successfully after Phase 6 has already published the release.

This phase is a post-publish validation follow-up. It does not redesign the Phase 6 release pipeline, does not add a new distribution source, does not change the bootstrap contract from Phase 5, and does not absorb the separate corporate-firewall validation backlog.

</domain>

<decisions>
## Implementation Decisions

### Execution venue
- **D-01:** Phase `06.1` runs as a CI-only fresh-runner matrix across all five supported targets.
- **D-02:** For this phase, "fresh machine" means a clean hosted runner with an empty `pure-simdjson` cache, not a manually prepared workstation.

### Source ladder coverage
- **D-03:** The normal validation path uses live R2 artifacts on all five supported targets.
- **D-04:** GitHub Releases fallback must also be validated, but only on a smaller representative subset instead of the full five-target matrix.

### Pass evidence
- **D-05:** Every validation run must begin from an empty cache and then exercise the real bootstrap path through `NewParser() -> Parse([]byte("42")) -> Root().GetInt64()`.
- **D-06:** A pass requires asserting that the expected platform cache file was created in the bootstrap cache layout after download.
- **D-07:** Unix targets must also assert that the created cache directory keeps the locked `0700` permission contract from Phase 5.

### Trigger model
- **D-08:** Phase `06.1` ships as a separate rerunnable workflow keyed by a published tag or version, not as an inline extension of `release.yml`.
- **D-09:** The public-validation workflow must stay outside the publish workflow so operators can rerun it against an already-published release without rebuilding or republishing artifacts.
- **D-10:** Manual scripts and runbook steps may support the implementation, but the canonical operator entrypoint for this phase is the dedicated rerunnable workflow.

### the agent's Discretion
- Exact choice of the smaller representative subset used for forced GitHub Releases fallback validation.
- Exact workflow trigger shape (`workflow_dispatch`, reusable workflow input, or equivalent) as long as it is rerunnable by published tag/version.
- Exact assertion mechanism for cache-path and permission checks, as long as the evidence is explicit and machine-verifiable in CI logs.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `.github/workflows/release.yml`: already defines the supported runner shapes and platform grouping used by the release path.
- `scripts/release/run_go_packaged_smoke.sh`: provides the exact loopback-mirror smoke pattern for proving the real bootstrap path without `PURE_SIMDJSON_LIB_PATH`.
- `tests/smoke/go_bootstrap_smoke.go`: minimal consumer proof that `NewParser()` can bootstrap and parse `42`.
- `internal/bootstrap/url.go`: single source of truth for platform names, cache filenames, R2 keys, and GitHub asset naming.
- `internal/bootstrap/bootstrap.go`: exposes the env knobs needed to force R2-only or GitHub-fallback behavior during live validation.
- `cmd/pure-simdjson-bootstrap/verify.go`: optional reusable primitive if planning later chooses to add post-download checksum re-verification beyond the currently locked pass criteria.

### Established Patterns
- Release publication and post-publish public validation are deliberately separated; `docs/releases.md` ends at successful publish and leaves live bootstrap validation to Phase `06.1`.
- The bootstrap contract already treats R2 as primary and GitHub Releases as fallback; Phase `06.1` should validate that ladder, not invent a new one.
- Cache layout and permissions are source-controlled behavior from Phase 5, so the validation phase should assert them explicitly rather than treating parse success alone as sufficient.

### Integration Points
- A new rerunnable workflow under `.github/workflows/` should become the canonical Phase `06.1` entrypoint.
- The workflow should drive the public bootstrap path against already-published tags or versions and record explicit cache-path / permission evidence in CI output.
- The implementation should reuse the existing Go bootstrap smoke and bootstrap package env controls rather than adding a second end-to-end test harness.

</code_context>

<specifics>
## Specific Ideas

- The five-target coverage should be CI-runner based, not workstation based.
- R2 is the main public path and should be exercised everywhere; GitHub Releases fallback only needs a narrower representative proof.
- A passing run is more than "parse worked": it should prove empty-cache start, bootstrap-created cache state, and unix cache permissions.
- The workflow should be rerunnable against a published tag/version without reopening the publish pipeline.

</specifics>

<deferred>
## Deferred Ideas

- Live corporate-firewall or proxy-specific validation remains separate from Phase `06.1`; it is already tracked as backlog follow-up work rather than part of this phase.
- A stronger post-download proof that also runs the bootstrap CLI `verify` command was considered but not locked for this phase.
- Automatic chaining from `release.yml` into the Phase `06.1` workflow was considered but deferred in favor of an independent rerunnable validation workflow.

</deferred>
