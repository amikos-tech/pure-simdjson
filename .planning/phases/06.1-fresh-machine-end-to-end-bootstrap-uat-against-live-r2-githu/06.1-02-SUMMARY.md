# Phase 06.1 Plan 02 Summary

**Added a dedicated rerunnable hosted-runner workflow for public bootstrap validation plus a repo-local contract test that locks the workflow shape and wrapper expectations against drift**

## Accomplishments

- Created `.github/workflows/public-bootstrap-validation.yml` as a `workflow_dispatch` workflow with required `version` input and concurrency `public-bootstrap-validation-${{ inputs.version }}`.
- Added separate `validate-r2` and `validate-gh-fallback` jobs, both with `shell: bash`, `timeout-minutes: 20`, and deterministic matrix membership.
- Implemented the dual-checkout pattern: current branch at repo root for workflow-owned tooling, requested tag into `target-src/` for validation of the published source tree.
- Added fail-fast target-source validation of `target-src/go.mod`, `target-src/tests/smoke/go_bootstrap_smoke.go`, and `target-src/internal/bootstrap/version.go` before toolchain setup.
- Added `scripts/release/test_public_bootstrap_validation_contracts.py` so the workflow pins, matrix membership, tag checkout model, and wrapper contract stay grep-testable and locally enforced.

## Decisions

- Kept the workflow manual-only and separate from `release.yml` so post-publish validation stays rerunnable without rebuilding or republishing artifacts.
- Used the smaller fallback subset `linux/amd64`, `darwin/arm64`, and `windows/amd64` because it covers each OS family once while the R2 job remains the full compatibility sweep.
- Reused pinned action SHAs from the release workflow so Phase `06.1` does not quietly widen the supply-chain surface.
