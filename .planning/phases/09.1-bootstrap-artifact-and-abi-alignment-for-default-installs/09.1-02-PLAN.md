<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| local source -> remote tag | Tag publication mutates the public release namespace. |
| release workflow -> public artifacts | CI publishes immutable R2 and GitHub Release assets. |
| public artifacts -> default install | Fresh `NewParser()` calls download, verify, cache, load, and ABI-check published artifacts. |
| validation evidence -> public docs | Docs may claim default-install readiness only after validation passes. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09.1-04 | Tampering | tag source | mitigate | Strict readiness and `release.yml` both require the tag commit to be anchored on `origin/main`. |
| T-09.1-05 | Elevation of Privilege | publication path | mitigate | CI is the only publish path; no hand uploads or bypasses are allowed. |
| T-09.1-06 | Spoofing | default-install validation | mitigate | Public validation checks out the published tag into `target-src`, clears cache/env state, and validates R2 plus GitHub fallback. |
| T-09.1-07 | Repudiation | release outcome | mitigate | Record strict readiness, release workflow, and validation workflow evidence in `09.1-RELEASE-VALIDATION.md`. |
| T-09.1-08 | Tampering | docs claims | mitigate | Docs hedge removal depends on recorded validation evidence and grep checks reject pending-release language afterward. |
</threat_model>
