# Phase 09.1 Research: Bootstrap artifact and ABI alignment

## Research Complete

Phase 09.1 is release-state work, not parser feature work. The current source pins `internal/bootstrap.Version` to `0.1.0` while the Go and Rust ABI constants are `0x00010001`, so a fresh default install can download a published artifact with the old ABI and fail `NewParser()` at the ABI gate. The plan should make the release candidate source self-consistent for `v0.1.2`, add a release-readiness guard that catches this class of drift before tagging, and make post-publish public bootstrap validation the completion gate.

## Current Repository Reality

- `internal/bootstrap/version.go` is still `const Version = "0.1.0"`.
- `internal/bootstrap/checksums.go` intentionally keeps `Checksums` empty in source; production bootstrap resolves expected digests from published `SHA256SUMS`.
- `internal/ffi/types.go` defines `ABIVersion uint32 = 0x00010001`.
- `src/lib.rs` defines `PURE_SIMDJSON_ABI_VERSION: u32 = 0x0001_0001`.
- `docs/releases.md` is the authoritative release runbook. It states the supported path is main commit -> strict readiness gate -> annotated tag -> CI publish, and `release.yml` validates that the tag commit is anchored on `origin/main`.
- `scripts/release/check_readiness.sh --strict --version <semver>` currently checks `bootstrap.Version`, Cargo metadata, `Cargo.lock`, workflow/runbook presence, and ancestry on `origin/main`.
- The context file references `scripts/release/assert_prepared_state.py` and `scripts/release/update_bootstrap_release_state.py`, but those files are not present in the current repo. Planning must not depend on them.
- `.github/workflows/public-bootstrap-validation.yml` already validates a published version across full R2 matrix and a smaller GitHub fallback matrix. It checks out `refs/tags/v<version>` into `target-src` and verifies `target-src/internal/bootstrap/version.go` before running smoke tests.

## Recommended Implementation Shape

### Pattern 1: Treat source release state as the pre-tag artifact contract

Update the release candidate source to `bootstrap.Version == "0.1.2"` and keep `Checksums` empty. The release workflow publishes `SHA256SUMS`; runtime bootstrap consumes those published digests. Do not reintroduce generated source checksum tables as a release dependency.

### Pattern 2: Add an explicit ABI release-state gate

Add a small standard-library Python helper, for example `scripts/release/check_bootstrap_abi_state.py`, and call it from `check_readiness.sh --strict`. The helper should:

- parse `internal/bootstrap/version.go`;
- parse Go `internal/ffi/types.go` ABI;
- parse Rust `src/lib.rs` ABI;
- require Go and Rust ABI constants to match;
- enforce a small policy table mapping ABI values to the minimum release version known to carry that ABI, currently `0x00010000 -> 0.1.0` and `0x00010001 -> 0.1.2`;
- optionally require `--version` to match `bootstrap.Version` so the helper can produce one focused diagnostic.

This is more accurate than the naive ABI-major-vs-semver-major comparison discussed in context: ABI `0x00010001` and semver `0.1.2` are expected to differ numerically. The invariant is not equal major numbers; the invariant is that the bootstrap pin names a release that carries the ABI expected by the wrapper.

### Pattern 3: Keep release publication human-gated

Plans should not run `git tag`, `git push`, or publish artifacts autonomously. The executable plan should include a human checkpoint that requires:

- the release candidate commit is on `origin/main`;
- `bash scripts/release/check_readiness.sh --strict --version 0.1.2` passes from that commit;
- an annotated `v0.1.2` tag is created and pushed by the operator;
- `release.yml` succeeds;
- `public-bootstrap-validation.yml` is dispatched with `version=0.1.2` and passes.

If post-publish validation fails after the tag exists, recovery is a corrected `v0.1.3`; do not mutate or reuse `v0.1.2`.

### Pattern 4: Remove upcoming-release hedges only after validation

README and benchmark docs currently describe `v0.1.2` as upcoming-release evidence. That is correct before Phase 09.1 validation. Remove that hedge only after the published `v0.1.2` artifact path passes public bootstrap validation.

## Verification Architecture

- Unit-level: `python3 scripts/release/test_check_bootstrap_abi_state.py`.
- Source-state: `bash scripts/release/check_readiness.sh --strict --version 0.1.2` from the prepared main commit.
- Full local suite: `go test ./... && cargo test -- --test-threads=1 && make verify-contract`.
- Post-publish: dispatch `.github/workflows/public-bootstrap-validation.yml` with `version=0.1.2` and require full R2 matrix plus GitHub fallback matrix success.

## Threat Model Notes

- **Spoofing:** A local or imported library can mask default-install failure. Post-publish validation must clear `PURE_SIMDJSON_LIB_PATH` and use fresh cache directories, which the existing workflow already does.
- **Tampering:** A tag not anchored on `origin/main` could publish unreviewed state. `release.yml` and `check_readiness.sh --strict` both anchor on `origin/main`; keep both.
- **Repudiation:** Release evidence must point to workflow runs and the exact tag version. Summaries should record the release workflow URL and public-bootstrap-validation workflow URL.
- **Denial of service:** Failed validation after tag publication must not lead to overwriting artifacts; recover with `v0.1.3`.

## Out Of Scope

- Parser API changes.
- Materializer or benchmark tuning.
- CI-side automatic source rewrites for `version.go` or `checksums.go`.
- Reintroducing embedded generated checksum maps as the normal release path.
- Generic release abstraction for other projects.
