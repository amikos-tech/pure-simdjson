# Phase 09.1: Bootstrap artifact and ABI alignment for default installs - Context

**Gathered:** 2026-04-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Align the bootstrap-pinned artifact/version/checksum state with a published library that matches the current Go/native ABI (`0x00010001`), then revalidate the default-install path so `NewParser()` succeeds through the intended bootstrap contract. The current bootstrap pin (`internal/bootstrap/version.go`) is `"0.1.0"`, which ships ABI `0x00010000` (pre-Phase 8). The Go wrapper expects `0x00010001`, causing every fresh `NewParser()` call to fail the ABI gate.

This phase does not change the Go public API, add new parser features, tune the materializer, or produce new benchmark evidence. Phase 9 owns benchmark recalibration; Phase 10 owns PR benchmark regression signal.

</domain>

<decisions>
## Implementation Decisions

### Release version label
- **D-01:** Target `v0.1.2` — not a separate `v0.1.1` patch. Phase 9 already committed all evidence paths under `v0.1.2` (`testdata/benchmark-results/v0.1.2/`, `docs/benchmarks/results-v0.1.2.md`, CHANGELOG Unreleased entries). Decoupling to `v0.1.1` would require renaming all of those paths with no meaningful benefit to consumers.
- **D-02:** The mechanical change is a single file: bump `internal/bootstrap/version.go` from `"0.1.0"` to `"0.1.2"`. The `checksums.go` map remains empty at source; the bootstrap resolves SHA-256 from the published `SHA256SUMS` file at runtime.

### Release scope bundling
- **D-03:** Bundle the bootstrap/ABI fix with Phase 9's benchmark docs, README positioning, and CHANGELOG entries into a single `v0.1.2` tag. Phase 9 PR #21 is merged — the coordination dependency that makes bundling risky is dissolved. A single tag delivers both the ABI-aligned default-install path and the post-Phase-8 benchmark story simultaneously.
- **D-04:** Promote the CHANGELOG `Unreleased` section to a dated `v0.1.2` entry as part of the release-prep commit. Do not create a separate CHANGELOG section; the Unreleased content already covers both deliverables.

### Default-install validation gate
- **D-05:** After `release.yml` publishes the `v0.1.2` artifacts, dispatch `public-bootstrap-validation.yml` (the Phase 06.1 workflow) targeting `v0.1.2`. Treat a green 5-platform matrix (linux/amd64, linux/arm64, darwin/amd64, darwin/arm64, windows/amd64) as the go/no-go signal before declaring the release done.
- **D-06:** The dispatch is post-tag. If `public-bootstrap-validation.yml` fails after the `v0.1.2` tag is pushed, the recovery path is a corrected `v0.1.3` — consistent with how the `pure-*` family handles post-tag issues. No pre-release staging slot required.
- **D-07:** The local developer quick-check (clear `PURE_SIMDJSON_LIB_PATH`, run `go test ./... -run TestNewParser` or similar) is an acceptable sanity step before triggering the release workflow, but it is not the acceptance gate.

### ABI-drift prevention
- **D-08:** Extend `check_readiness.sh --strict` with an ABI-major cross-check assertion: extract the major nibble from `ffi.ABIVersion` (top 16 bits of `0x00010001` = `1`) and assert it equals the semver major of `bootstrap.Version` (semver major of `"0.1.2"` = `0`). If the encoding convention changes in a future major version bump, the assertion will catch it at tag time.

  > **Note on current encoding:** `ABIVersion 0x00010001` (major nibble = `1`) vs bootstrap semver major `0` — these are not equal under the naive comparison. The real invariant to assert is that when `ffi.ABIVersion` is bumped, `bootstrap.Version` must also be bumped to a newly published release carrying that ABI. The assertion should flag when `bootstrap.Version` has not been updated since the last ABI bump — a `git log`-based check or a constant-version pinning check in the script is acceptable.

- **D-09:** Add a trivial Go build-time canary in a new file `internal/bootstrap/abi_assertion.go` that uses `//go:linkname` or a blank `var` assignment to bind together `bootstrap.Version` and `ffi.ABIVersion`, failing `go build` if the bootstrap pin is stale relative to the ABI expectation. This is a zero-maintenance complement to D-08 that catches drift during development, not just at release time.
- **D-10:** Fully automating the bootstrap-pin rewrite in CI (`update_bootstrap_release_state.py`) is deferred — out of scope for Phase 09.1. The manual step + `check_readiness.sh` gate is the v0.1 approach.

### Carrying forward from Phase 9
- Phase 9 D-10: `v0.1.2` label confirmed for benchmark snapshot and release.
- Phase 9 D-15: Phase 09.1 performs release/bootstrap artifact alignment before any tag is cut.
- Phase 9 D-17: README benchmark language framed as "upcoming-release claim" until Phase 09.1 validates — after `public-bootstrap-validation.yml` passes, the upcoming-release hedge can be removed.

### Claude's Discretion
- Exact form of the `check_readiness.sh` ABI-major assertion (git log comparison, constant grep, or semver parse).
- Exact implementation of the Go build-time canary (`abi_assertion.go`) — a `const` cross-check, a `func init()` panic guard, or a blank `var` assignment are all acceptable.
- Exact wording for CHANGELOG `v0.1.2` dated section, as long as it covers both the bootstrap/ABI fix and the Phase 9 benchmark positioning notes.

</decisions>

## Existing Code Insights

### Reusable Assets
- `public-bootstrap-validation.yml` + `scripts/release/run_public_bootstrap_smoke.sh` — battle-tested from Phase 06.1; covers all 5 platforms with real R2 + GitHub fallback; no new files needed for the validation gate
- `scripts/release/update_bootstrap_release_state.py` — rewrite `version.go` and `checksums.go` from manifest entries; use this on the release-prep branch to produce the correct `"0.1.2"` pin
- `scripts/release/check_readiness.sh --strict` — already validates `bootstrap.Version == tag version`; D-08 extends this with an ABI-major cross-check
- `internal/bootstrap/export_test.go` — test seam that re-exports `ResetBootstrapFailureCacheForTest` and related; useful for integration test coverage of the updated bootstrap path

### Established Patterns
- `internal/bootstrap/version.go` is the single source of truth for the bootstrap pin; never set via `ldflags -X` (D-06 from Phase 6)
- `checksums.go` map is intentionally empty in source; production bootstrap resolves SHA-256 from `SHA256SUMS` published under the release tag
- Release-prep commits must update `version.go`, `checksums.go`, and `CHANGELOG.md` on a `release-prep/v<version>` branch before the tag is created (Phase 6 pattern)
- `testmain_test.go` seeds `PURE_SIMDJSON_LIB_PATH` to `target/release/<libname>` when the Cargo artifact exists, so normal test runs bypass network; integration tests that exercise the bootstrap path must explicitly clear this env var

### Integration Points
- `library_loading.go` — the 4-stage resolution chain: env override → cache hit → `BootstrapSync` → cache hit after bootstrap; Stage 3 is what this phase validates
- `parser.go:44-46` — ABI check that fails when `v0.1.0` artifacts are loaded; this is the observable failure mode Phase 09.1 fixes
- `CHANGELOG.md` — `Unreleased` section (containing Phase 9 benchmark-positioning notes) gets promoted to `## [0.1.2]` dated entry during release-prep

</code_context>

<specifics>
## Specific Ideas

- The core change is one file (`version.go`): bump `"0.1.0"` to `"0.1.2"`. Everything else in this phase is process/validation work to ensure the release is correct and that future ABI bumps don't cause the same lag.
- The Go build-time canary (`abi_assertion.go`) was specifically chosen as a complement to the `check_readiness.sh` gate because it fires during `go build`, not only at release time — catching any contributor who bumps `ffi.ABIVersion` without updating the bootstrap pin before a PR is even opened.
- After `public-bootstrap-validation.yml` passes on the `v0.1.2` tag, the README benchmark section should have its "upcoming-release" hedge removed — that was the explicit Phase 9 D-17 precondition.

</specifics>

<deferred>
## Deferred Ideas

- Fully automating the bootstrap-pin rewrite in release-prep CI (`update_bootstrap_release_state.py` called from a CI workflow step that writes and commits `version.go`) — reduces the manual step but is out of scope for Phase 09.1.
- Post-publish CI ABI consistency job (dlopen on published artifacts across all 5 platforms) — high complexity and chicken-and-egg for new releases; belongs in a dedicated hardening or ops phase.
- Pre-release artifact staging slot for zero-risk pre-tag validation — deviates from the established `pure-*` release runbook and is overkill given the `v0.1.3` recovery path is well-defined.

</deferred>
