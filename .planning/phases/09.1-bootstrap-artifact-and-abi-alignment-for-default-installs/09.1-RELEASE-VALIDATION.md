## Recovery Notes

If post-publish validation fails after v0.1.4 exists, recover with a corrected v0.1.5; do not mutate or reuse v0.1.4.

The Phase 09.1 release was originally planned for v0.1.2 (Plan 02). Two failed publication cycles required corrective patches rather than mutating the failing tags:

- **v0.1.2 → v0.1.3 (PR #23 "Recover v0.1.3 release publication"):** Linux release artifacts hid non-public dynamic symbols so the exported surface stays limited to the public `pure_simdjson_*` ABI; macOS release smoke no longer reports parser diagnostic string cleanup as an untracked native allocator free.
- **v0.1.3 → v0.1.4 (PR #24 "Recover v0.1.4 release build"):** Linux release builds no longer depend on custom `std::basic_string` allocator behavior that is rejected by the manylinux libstdc++ toolchain. Bootstrap default-install was re-pinned to `0.1.4`.

The no-mutate-tag policy held under stress: v0.1.2 and v0.1.3 tags remain in place as evidence of the failed states; v0.1.4 is the validated default-install version.

CHANGELOG.md `[0.1.3]` and `[0.1.4]` sections capture the per-recovery details for downstream consumers.
