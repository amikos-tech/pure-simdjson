# Cross-AI Plan Review - Phase 09.1

## Gemini Review

This review evaluates the implementation plans for **Phase 09.1: Bootstrap artifact and ABI alignment for default installs**.

### Summary

The proposed plans provide a comprehensive and robust strategy for resolving the ABI mismatch between the Go wrapper and the published bootstrap artifacts. By introducing both a pre-tag validation script (`check_bootstrap_abi_state.py`) and a Go build-time canary (`abi_assertion.go`), the plans effectively prevent future "silent" drift between the ABI version and the bootstrap release pin. The two-wave approach correctly separates source preparation and local verification from the sensitive, human-gated release and public validation steps.

### Strengths

- **Defense in Depth:** The combination of a Python-based readiness gate and a Go compile-time assertion ensures that ABI drift is caught both by the release operator and by developers during normal testing.
- **Pattern Adherence:** The plans strictly follow the established `pure-*` release runbook (tag-driven, CI-only publish, R2 + GitHub fallback).
- **Self-Documenting Recovery:** Explicitly documenting the `v0.1.3` recovery path in the validation artifact mitigates the risk of "fixing" a published tag, which would break the immutability contract.
- **Testing for the Tester:** Including a dedicated unit test suite for the new ABI validator (`test_check_bootstrap_abi_state.py`) ensures the gate itself is reliable.
- **Hedge Management:** The strategy for removing "upcoming-release" language only after public validation evidence is recorded demonstrates high discipline in maintaining documentation accuracy.

### Concerns

- **Regex Fragility (LOW):** The Python validator relies on regular expressions to parse Go and Rust source files. While suitable for a release script, these could break if the formatting of `internal/ffi/types.go` or `src/lib.rs` changes significantly, such as adding line breaks or comments between tokens.
- **CI Dependency (LOW):** The successful completion of Plan 02 depends entirely on the availability and correctness of `release.yml` and the R2/GitHub distribution pipeline. This is an inherent risk of the project's architecture, mitigated here by the post-publish validation gate.

### Suggestions

- **Validator Robustness:** In `check_bootstrap_abi_state.py`, consider using a slightly more flexible regex or a simple line-by-line scanner that strips comments/whitespace before matching to ensure the ABI constant can be found even if formatted differently.
- **Canary Naming:** The name `abiVersionForBootstrapVersion_0_1_2` in the Go canary is specific. Ensure that the task instructions for future ABI bumps explicitly mention updating this constant name to prevent "name drift" where the constant name says `0_1_2` but the value represents a later version.

### Risk Assessment

**Overall Risk: LOW**

The phase is primarily focused on configuration alignment and process enforcement. It does not touch the core parser logic or the materializer. The most dangerous part, tagging and publishing, is guarded by a human checkpoint and a validated CI workflow. The use of a compile-time canary in Go is a particularly low-risk, high-reward safety measure.

### Plan-Specific Notes

#### Plan 01 (Source & Readiness)

The logic for the policy table in `check_bootstrap_abi_state.py` (mapping ABI to minimum version) is the correct way to handle the numerical difference between ABI `0x00010001` and SemVer `0.1.2`. This is a significant improvement over the naive "major nibble" comparison originally discussed in the context.

#### Plan 02 (Release & Validation)

The requirement to dispatch `public-bootstrap-validation.yml` and record its outcome before updating documentation is excellent. It ensures that the README only makes claims that have been empirically verified against the live production artifacts.

## Claude Review

### Plan Review: Phase 09.1 - Bootstrap Artifact and ABI Alignment

#### Plan 01 (`09.1-01-PLAN.md`) - Release-Candidate Source State

##### Summary

Plan 01 is tightly scoped and well-structured. It correctly identifies that the entire phase is bottlenecked by one mechanical change (`version.go` bump) surrounded by preventive tooling. The TDD-first ABI checker, compile-time canary, and strict-mode readiness integration form a defense-in-depth chain from development to tag time. Task sequencing is mostly correct but there is one ordering issue in Task 1's acceptance criteria that needs attention.

##### Strengths

- **Bidirectional canary design:** Using two array-size constant expressions instead of a runtime panic guard (`func init()`) means ABI drift fails at `go build`, not `go test`, and adds zero binary overhead.
- **Policy table over numeric comparison:** `ABI_MINIMUM_VERSION` correctly captures the human invariant (which release first ships ABI X) rather than the brittle numerical one from CONTEXT.md D-08.
- **TDD ordering for the checker:** Writing `test_check_bootstrap_abi_state.py` before the implementation produces a well-specified contract, and the test invokes the checker as a subprocess rather than importing private functions.
- **`--repo-root` defaulting:** Inferring the repo root from the script path makes the checker usable both locally and in CI without caller configuration.
- **Acceptance criteria are executable:** Nearly every criterion is a runnable shell or Python command, not a prose description.
- **Correct non-regression guard:** `bash scripts/release/check_readiness.sh` (basic mode, no `--strict`) is the final auto-verification step, appropriate since the prepared commit is not yet on `origin/main`.

##### Concerns

**HIGH: Task 1 acceptance criterion depends on Task 2 output**

Task 1 includes:

```sh
python3 scripts/release/check_bootstrap_abi_state.py --version 0.1.2
```

The note "after Task 2" is acknowledged inline, but the acceptance criteria block for Task 1 includes this command. If the executor runs Task 1 acceptance checks before Task 2, which is the normal sequential flow, this criterion will fail because `version.go` still reads `"0.1.0"`. The criterion should be deferred to Task 2's acceptance block, or the note should be elevated to a separate post-Task-2 verification step.

**MEDIUM: CHANGELOG date is hardcoded as 2026-04-24 but execution is 2026-04-25**

The plan specifies `## [0.1.2] - 2026-04-24`. The review was run on 2026-04-25. The executor will write a dated entry that is one day in the past. This is minor because Keep-a-Changelog does not require dates to match the commit timestamp, but the acceptance criterion `rg '^## \[0\.1\.2\] - 2026-04-24' CHANGELOG.md` will fail if the executor uses the current date. Either the date should be flexible in the criterion or explicitly noted as the intended date.

**MEDIUM: `checksums.go` assertion is format-sensitive**

```python
assert 'var Checksums = map[string]string{\n' in text and text.count('<sha256>') == 5
```

This assertion will break if `gofmt` or the executor writes `var Checksums = map[string]string{` with trailing spaces or different newline handling. A more robust check would grep for `<sha256>` occurrences in the file without the exact opening brace format.

**LOW: `! rg 'v0\.1\.0' docs/bootstrap.md` is aggressive**

`docs/bootstrap.md` currently contains `TAG=v0.1.0` in the cosign verification example. Replacing all `v0.1.0` with `v0.1.2` is correct. However, the cosign release runbook examples in `docs/releases.md` are not in the files_modified list. If there are cross-doc forward references, the blanket negation check could create false confidence about completeness.

**LOW: TDD-tagged task for a non-code artifact**

Task 1 is `tdd="true"` but it produces Python scripts, not Go code. The execute-plan TDD protocol is designed for compiled languages. The executor should understand that TDD here means "write the test file first, verify it fails, then write the checker."

##### Suggestions

- Move `python3 scripts/release/check_bootstrap_abi_state.py --version 0.1.2` from Task 1's acceptance criteria to Task 2's acceptance criteria, or add it to the plan-level verification block.
- Relax the CHANGELOG date criterion, for example `rg '^## \[0\.1\.2\] - 202' CHANGELOG.md`, or explicitly document "use 2026-04-24 regardless of execution date."
- Replace the fragile Python inline assertion on `checksums.go` with a count-based check such as `rg '<sha256>' internal/bootstrap/checksums.go | wc -l | grep -q '^5$'`.
- Add `docs/releases.md` to the `files_modified` list if the examples there reference `v0.1.0`, or explicitly confirm it does not need updating.

##### Risk Assessment: LOW

Both tasks are deterministic source edits with no external dependencies. The ABI canary is compile-time and provably correct for the bidirectional case. The main risk is the acceptance-criteria ordering issue, which is a test-runner concern rather than a correctness concern.

#### Plan 02 (`09.1-02-PLAN.md`) - Release Publication and Hedge Removal

##### Summary

Plan 02 is correctly structured as a human-gated release operation. The checkpoint/auto task split is appropriate because the human gate enforces the "CI is the only publish path" invariant from the release skill. Task 2 (hedge removal) is cleanly conditioned on Task 1 approval. There is one significant gap: a phantom test file referenced in acceptance criteria that does not exist and is not scheduled to be created.

##### Strengths

- **`autonomous: false` is correct:** The plan correctly refuses to tag or push autonomously. Tagging is irreversible and must be operator-controlled.
- **Recovery path is explicit:** The mandatory sentence about recovering with `v0.1.3` ensures the executor records and the operator understands the no-mutation invariant.
- **Hedge removal is strictly post-validation:** Task 2 is conditioned on Task 1 approval, and acceptance criteria explicitly check that "upcoming-release" language is gone.
- **Immutability guard on CI artifacts:** `git diff --exit-code -- .github/workflows/release.yml .github/workflows/public-bootstrap-validation.yml testdata/benchmark-results/v0.1.2` correctly verifies that the executor did not accidentally edit CI config or benchmark evidence.
- **Dual-band validation requirement:** Requiring both full R2 matrix and GitHub fallback subset is the established Phase 06.1 contract and provides appropriate coverage.

##### Concerns

**HIGH: `test_public_bootstrap_validation_contracts.py` is a phantom file**

Task 2's acceptance criteria and verification block both reference:

```sh
python3 scripts/release/test_public_bootstrap_validation_contracts.py
```

This file does not exist in the repository, is not in `files_modified` for either plan, and is not mentioned in the research, patterns, or validation artifacts. The executor will either create it, which is undefined scope, or the acceptance check will fail. This needs to be removed from the acceptance criteria, replaced with an existing test, or explicitly added to `files_modified` with creation instructions.

**MEDIUM: `docs/bootstrap.md` changes in Task 2 may conflict with Plan 01 Task 2**

Plan 01 Task 2 modifies `docs/bootstrap.md` by updating `v0.1.0` examples to `v0.1.2`. Plan 02 Task 2 also modifies `docs/bootstrap.md` by adding a `## Current Release` section. This is fine across plans, but Plan 02 should make the dependency on the post-Plan-01 state explicit to avoid conflicting or redundant content.

**MEDIUM: Release validation document is write-only**

The `09.1-RELEASE-VALIDATION.md` acceptance criteria check that the document exists and has specific headings, but Task 2's verification block does not re-read or cross-reference the validation document when removing hedges. The acceptance criteria for Task 2 could pass even if the release validation document contained a failure. Consider adding a check that `09.1-RELEASE-VALIDATION.md` contains evidence of passing validation.

**LOW: `docs/benchmarks/results-v0.1.2.md` release boundary language**

The task says to replace pending release boundary language with language saying `v0.1.2` has passed post-publish public bootstrap validation. However, the artifacts block requires the exact phrase `validated default-install path`, while the action instructions do not explicitly require that phrase. The executor might write semantically correct language that fails the grep.

**LOW: No local quick-check for the ABI gate itself**

Per CONTEXT.md D-07, a local developer quick-check, such as clearing `PURE_SIMDJSON_LIB_PATH` and running a focused `TestNewParser`, is an acceptable sanity step before triggering the release workflow. Plan 02 does not include this as an intermediate verification step before the human checkpoint. It is not required, but it would be useful.

##### Suggestions

- Remove or create `test_public_bootstrap_validation_contracts.py`. If it should exist, add it to `files_modified` with creation instructions. If it does not need to exist, remove it from all acceptance criteria and verification blocks.
- In Task 2 verification, add a grep against `09.1-RELEASE-VALIDATION.md` for evidence of passing validation, for example `rg 'PASSED|validated version|all jobs pass' .planning/phases/.../09.1-RELEASE-VALIDATION.md`.
- Add an explicit note to read the Plan 01 state of `docs/bootstrap.md` before adding the `## Current Release` section.
- Consider adding the local quick-check as an optional pre-checkpoint step or noting it in checkpoint instructions.

##### Risk Assessment: MEDIUM

The phantom `test_public_bootstrap_validation_contracts.py` reference is a concrete blocker for Task 2's acceptance gate. It will either fail if the executor does not create the file, or introduce undefined scope if it does. All other concerns are minor. The underlying release operations are sound given the established Phase 06.1 infrastructure.

### Cross-Plan Observations

- Sequencing is correct: Plan 01 is `autonomous: true` for source edits, and Plan 02 is `autonomous: false` for release operations. The `depends_on: ["09.1-01"]` chain is proper.
- The compile-time canary is the highlight of this phase. It shifts ABI-bootstrap drift detection from a release-time gate to a development-time compile failure.
- The policy table is better than D-08's naive comparison. The original context decision described asserting `ABI major == semver major`, which is numerically incorrect for `0x00010001` and `0.1.2`. The plans correctly follow the research instead.
- Wave 2 human gating aligns with the release-family convention: CI is the only publish path and tags must be anchored on `origin/main`.

## Consensus Summary

Both reviewers agree that the phase shape is sound: Plan 01 handles source-state alignment and drift prevention, while Plan 02 correctly separates irreversible release publication behind a human checkpoint. Both reviewers also agree that the policy-table approach for ABI-to-release mapping and the Go compile-time canary are strong design choices.

### Agreed Strengths

- The plans preserve the release boundary: source preparation is automated, while tag creation and publication remain human-gated.
- ABI drift is addressed in depth through both a release-readiness checker and a development-time compile canary.
- The post-publish validation gate prevents public docs from claiming default-install readiness before live artifacts are proven.
- The recovery path for a failed published release is explicit: publish a corrected `v0.1.3` rather than mutating `v0.1.2`.

### Agreed Concerns

- The Python checker should be robust enough to tolerate reasonable formatting changes in Go and Rust ABI constants.
- Plan 02 relies on release and bootstrap validation infrastructure, so the validation evidence should be checked directly before docs hedges are removed.
- Claude identified two concrete execution blockers that should be addressed before execution: Task 1 includes a post-Task-2 success check in its own acceptance criteria, and Plan 02 references `scripts/release/test_public_bootstrap_validation_contracts.py` even though that file does not currently exist.

### Divergent Views

- Gemini rated overall phase risk as LOW because the release operation is human-gated and the implementation does not touch parser internals.
- Claude rated Plan 02 risk as MEDIUM due to the missing `test_public_bootstrap_validation_contracts.py` acceptance dependency. This is the highest-priority actionable review item.
