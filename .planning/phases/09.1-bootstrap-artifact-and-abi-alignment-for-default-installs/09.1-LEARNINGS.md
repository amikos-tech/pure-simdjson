# Phase 09.1 Learnings: bootstrap-artifact-and-abi-alignment-for-default-installs

## Decisions

### ABI policy table over major-bit-to-semver-major comparison
The release-state checker enforces a concrete `ABI_MINIMUM_VERSION` table mapping ABI integer values to minimum semver versions (e.g. `0x00010000 -> 0.1.0`, `0x00010001 -> 0.1.2`).

**Rationale:** Comparing ABI major bits to semver major numbers is structurally weaker — it conflates two unrelated numbering schemes and silently passes when both happen to align. An explicit table makes the invariant auditable and fails closed on the unknown-ABI case.
**Source:** 09.1-01-PLAN.md, 09.1-01-SUMMARY.md (Decisions Made)

---

### Checksums map empty in source; production digests resolved from published SHA256SUMS
`internal/bootstrap/checksums.go` ships with an empty `Checksums` map. Production runtime verification reads the published `SHA256SUMS` artifact from the release; tests inject digests via package-level overrides.

**Rationale:** Embedding per-version digests in source means every recovery patch (v0.1.3, v0.1.4) would force a source-state regeneration step on the prep branch. Empty-in-source plus published-SHA256SUMS means recovery only requires advancing the version constants and rebuilding, not regenerating committed digest tables.
**Source:** 09.1-01-SUMMARY.md (Decisions Made), 09.1-02-SUMMARY.md (Recovery Cycle Detail)

---

### Compile-time Go ABI canary as a bidirectional array-length equality
`internal/bootstrap/abi_assertion.go` declares two arrays whose lengths reference the bootstrap version constant and `ffi.ABIVersion` respectively. `go test ./internal/bootstrap` fails to compile on drift.

**Rationale:** Runtime-only ABI checks fail late (parser instantiation). A compile-time canary tied to package init means drift is caught by `go vet` / `go build` before any test runs, which is essential for recovery cycles where every iteration needs fast local feedback.
**Source:** 09.1-01-SUMMARY.md (key-decisions)

---

### Tag mutation is forbidden; recovery proceeds via new patch versions
When a published tag fails (release.yml fails or post-publish validation fails), the path forward is the next patch version, not a re-tag. v0.1.2 and v0.1.3 tags remain in place as evidence of failed states.

**Rationale:** A mutated tag breaks every consumer that already pinned to the original SHA. The cost of a never-consumed v0.1.2 sitting in tag history is zero; the cost of a silently re-tagged v0.1.2 with different artifact bytes is catastrophic.
**Source:** 09.1-02-PLAN.md (must_haves.truths), 09.1-02-SUMMARY.md (Decisions Made)

---

### Recovery scope bounded to the publish pipeline; parser hot path frozen
v0.1.3 and v0.1.4 changed only build/packaging correctness: macOS allocator-cleanup telemetry attribution, Linux symbol visibility, manylinux libstdc++ `std::basic_string` allocator behavior. `src/native/simdjson_bridge.cpp` was touched but only in the materializer support paths, not the parse/extract hot path.

**Rationale:** Recovery cycles already pay a per-iteration tag/release cost. Mixing parser-level changes into a recovery would invalidate the v0.1.2 benchmark baseline and require a fresh benchmark capture, which does not belong on the recovery critical path.
**Source:** 09.1-02-SUMMARY.md (Decisions Made), README.md:67

---

### v0.1.2 benchmark snapshot remains the validated baseline through v0.1.4
No re-benchmark for v0.1.3 or v0.1.4. The README cites `docs/benchmarks/results-v0.1.2.md` as the validated baseline and explicitly notes that the recovery releases targeted the publish pipeline rather than the parse/materialize hot paths.

**Rationale:** Re-running the full benchmark suite for each recovery patch is expensive and produces noise rather than signal when no parser code changed. The honest framing is "v0.1.4 ships the v0.1.2 parser bytes; v0.1.2 benchmark numbers apply".
**Source:** 09.1-02-SUMMARY.md (Decisions Made), README.md:67

## Lessons

### Originally-targeted version is not the shipped version when recovery happens
Plan 02 hardcoded `v0.1.2` throughout its must_haves, acceptance criteria, and evidence headings. The actual shipped version became `v0.1.4` after two recovery cycles. The plan's must_have artifacts had to be retroactively re-targeted.

**Context:** Plans that bake in a specific version label create brittle anchors. When recovery shifts the version, the plan reads as misaligned with the codebase even though the *intent* was satisfied. Future plans should phrase must_haves in version-agnostic terms (e.g. "the published validated version") and capture the actual shipped version in the SUMMARY/evidence file.
**Source:** 09.1-02-SUMMARY.md (Deviations from Plan), 09.1-UAT.md (gaps A and B)

---

### Recovery PRs default to fixing the failure mode and skip post-ship documentation
PR #23 and PR #24 each merged with version-pin advances and code fixes, but neither produced the Plan 02 must_have `09.1-RELEASE-VALIDATION.md` artifact and neither updated `README.md:67` from "upcoming-release" framing to validated-release framing. Both gaps surfaced only when retroactive UAT ran two days later.

**Context:** Recovery is reactive — engineers focus on what made the publish fail. Post-ship documentation steps that were on the original plan get implicitly re-scheduled to "after this works" and silently dropped. A concrete remediation: every recovery PR template should include a checklist item "post-ship doc update for [version]: needed | not needed".
**Source:** 09.1-UAT.md (Gaps section, root_cause for both gaps)

---

### Local strict-readiness pass is not equivalent to release.yml success
`bash scripts/release/check_readiness.sh --strict --version 0.1.2` passed locally before the v0.1.2 tag was pushed, yet release.yml failed. The two recovery cycles surfaced macOS placement-new, manylinux libstdc++ allocator behavior, and Linux symbol visibility — none of which the local strict-readiness gate exercises.

**Context:** Strict-readiness validates *source state* (versions, ABI consistency, ancestry). It is not a build / packaging / signing simulator. Treating local strict-readiness as a release predictor is a category error: the only true predictor is `release.yml` itself running on a real CI runner. Any future "is this ready to tag?" tooling must include a CI dry-run signal, not just source assertions.
**Source:** 09.1-02-SUMMARY.md (Recovery Cycle Detail), 09.1-01-SUMMARY.md (Issues Encountered: strict readiness reserved for prep main commit)

---

### Plan tasks of type `checkpoint:human-verify` can deliver across many PRs
Plan 02 declared a single `checkpoint:human-verify` task. Its outcome was actually delivered across PR #22 (initial publish attempt), PR #23 (first recovery), PR #24 (final shipping recovery), and PR #25 (post-ship gap closure). The task-count metric (1) is not a useful proxy for shipping cost.

**Context:** Human-verify checkpoints externalize the work to operator-driven CI runs, which can fail in unpredictable ways and snowball into multi-PR sequences. SUMMARY templates for human-verify tasks should explicitly enumerate "PRs that delivered this task" instead of assuming a single linear commit chain.
**Source:** 09.1-02-SUMMARY.md (Task Commits)

---

### CHANGELOG dated sections are the durable recovery record when validation evidence files lag
The Plan 02 must_have `09.1-RELEASE-VALIDATION.md` was missing for ~36 hours. During that window, the only durable record of what v0.1.3 and v0.1.4 actually fixed lived in `CHANGELOG.md` `[0.1.3]` and `[0.1.4]` dated sections plus the merged-PR titles ("Recover v0.1.3 release publication", "Recover v0.1.4 release build").

**Context:** Process artifacts (`*-RELEASE-VALIDATION.md`) are best-effort; user-facing artifacts (CHANGELOG) are mandatory. Putting per-recovery detail into CHANGELOG sections meant the recovery story survived the post-ship documentation drift. Future recovery work should commit a CHANGELOG entry in the same PR as the version-pin advance, not as a separate doc step.
**Source:** 09.1-02-SUMMARY.md (Recovery Cycle Detail), 09.1-UAT.md test 9

## Patterns

### ABI/version pin lock-step advance per recovery
Every recovery PR advances `internal/bootstrap/version.go`, the `abi_assertion.go` canary version label, and the `internal/bootstrap/checksums.go` per-version comments together in one commit. None of them moves alone.

**When to use:** Any recovery cycle on a release that pins both a semver and an ABI integer. The lock-step pattern means the compile-time canary either holds (recovery is internally consistent) or fails fast (the human forgot one of the three sites).
**Source:** 09.1-02-SUMMARY.md (key-files modified, by recovery)

---

### Tag-mutation-free recovery
On a failed publish, the failing tag remains. Recovery is a new patch version with its own annotated tag, its own dated CHANGELOG section, and its own validated artifacts. The failing tag is preserved as evidence, not deleted.

**When to use:** Any release pipeline where consumers can pin tags. The pattern trades a slightly busier tag list for an immutability guarantee that makes downstream pinning safe.
**Source:** 09.1-02-PLAN.md (must_haves.truths), 09.1-02-SUMMARY.md (key-decisions)

---

### Synthetic-source release-state tests
`scripts/release/test_check_bootstrap_abi_state.py` builds tiny synthetic source trees in `tempfile.TemporaryDirectory()` and runs the production checker against them — covering valid `0.1.2`, stale bootstrap pin, Go/Rust ABI mismatch, unknown-ABI policy, and version-mismatch cases.

**When to use:** Any release-state checker that parses minimal facts from real source files. Synthetic-source tests cover the failure-mode space without polluting the real repo and without a separate fixture directory.
**Source:** 09.1-01-SUMMARY.md (tech-stack patterns)

---

### Six-heading operator-recorded validation evidence
`{phase}-RELEASE-VALIDATION.md` is a structured artifact with six fixed headings: `Strict Readiness`, `Release Workflow`, `Public Bootstrap Validation`, `Outcome`, `Recovery Notes`, plus the document title. Each section captures a specific gate result with literal `PASSED` lines and run identifiers.

**When to use:** Any phase whose outcome depends on operator-driven CI runs that span more than one workflow. The fixed-heading template makes the evidence file `rg`-greppable for acceptance-criteria verification (Plan 02 used `rg` patterns to validate its own evidence file).
**Source:** 09.1-02-PLAN.md (acceptance_criteria), 09.1-RELEASE-VALIDATION.md

---

### Retroactive UAT to close post-ship documentation drift
Phase 09.1's UAT ran two days after v0.1.4 shipped, against the actual shipped reality. It surfaced two doc-only gaps (RELEASE-VALIDATION.md missing, README hedged) that the original Plan 02 acceptance gates would have caught at execute time but missed because of the recovery cycle drift.

**When to use:** Any phase that shipped through recovery cycles or had its scope shift mid-execution. Retroactive UAT against "what actually happened" closes the gap between what the plan promised and what the artifacts record.
**Source:** 09.1-UAT.md (entire workflow), 09.1-02-SUMMARY.md (Issues Encountered)

## Surprises

### v0.1.2 release.yml failed despite local strict-readiness passing
After Plan 01 prepared the source state and `check_readiness.sh --strict --version 0.1.2` passed locally, the v0.1.2 publication still failed because of macOS placement-new clash plus manylinux libstdc++ `std::basic_string` allocator behavior — both build/packaging-only failure modes invisible to the source-state checker.

**Impact:** Triggered the first recovery cycle (v0.1.3). Confirmed that local strict-readiness is a necessary but not sufficient pre-publish gate; full CI dry-run is the only true predictor of release.yml success.
**Source:** 09.1-02-SUMMARY.md (Recovery Cycle Detail, v0.1.2 row)

---

### v0.1.3 also failed despite the v0.1.2 fixes
After PR #23 landed v0.1.3 with fixes for the v0.1.2 failure modes, release.yml on v0.1.3 *still* failed because symbol-visibility audit and macOS native-string-cleanup telemetry attribution surfaced as additional gates. The recovery had to recover.

**Impact:** Triggered the second recovery cycle (v0.1.4). Demonstrated that publish-pipeline failure modes can stack: fixing one set surfaces the next set. Future planning should reserve recovery budget for at least 2 cycles, not assume the first patch will succeed.
**Source:** 09.1-02-SUMMARY.md (Recovery Cycle Detail, v0.1.3 row)

---

### Plan 02 had 1 task but shipped through 4 PRs
Plan 02 was scoped as a single `checkpoint:human-verify` task: publish + record validation evidence. The actual delivery sequence was PR #22 (initial attempt) -> PR #23 (v0.1.3 recovery) -> PR #24 (v0.1.4 final ship) -> PR #25 (post-ship doc gap closure). 1 task, 4 PRs, ~3 days elapsed.

**Impact:** Phase performance metrics (`task_count`, `duration`) are misleading for human-verify checkpoints. The Plan 01 SUMMARY recorded 4-minute duration; Plan 02 was effectively measured in PR count, not minutes.
**Source:** 09.1-02-SUMMARY.md (Performance, Task Commits)

---

### Post-ship documentation drift was invisible without retroactive UAT
The `09.1-RELEASE-VALIDATION.md` Plan 02 must_have was missing for ~36 hours after v0.1.4 shipped. README.md:67 contradicted shipped reality for the same window. Neither the recovery PR review process nor the merge-on-main signal flagged either gap. Only running `/gsd-verify-work 09.1` after the fact surfaced them.

**Impact:** Established retroactive UAT as a standard step after recovery cycles. Without it, phase artifacts can ship "complete" while the consumer-facing surface still hedges as pre-release.
**Source:** 09.1-UAT.md (Gaps section)

---

### `gsd-sdk query` handlers were unavailable in the execution environment
Plan 01 SUMMARY notes that the `gsd-sdk query` interface used by the GSD workflows was unavailable; summary and state updates had to be done manually. The same gap recurred in this Phase 09.1 closure session, where `gsd-sdk query init.progress` and `gsd-sdk query state` returned errors and `gsd-tools.cjs` had to be used as a fallback for a subset of operations.

**Impact:** Workflow files that target `gsd-sdk query` need a graceful fallback path or version-detect, otherwise routine planning operations require manual file authorship instead of structured tooling. Worth raising as a `/gsd-tools` infrastructure issue separately.
**Source:** 09.1-01-SUMMARY.md (Issues Encountered)
