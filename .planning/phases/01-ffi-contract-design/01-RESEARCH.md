## User Constraints (from CONTEXT.md) [VERIFIED: .planning/phases/01-ffi-contract-design/01-CONTEXT.md]

### Locked Decisions

### Already locked from project and research context
- **D-01:** `v0.1` stays DOM-based. On-Demand remains deferred to `v0.2`.
- **D-02:** Every `Parse` copies input into Rust-owned padded memory. No zero-copy-in path in `v0.1`.
- **D-03:** Iteration stays cursor/pull only. No Go callbacks across FFI.
- **D-04:** Number access stays split across distinct `int64`, `uint64`, and `float64` accessors with explicit overflow / precision-loss errors.
- **D-05:** Unsupported CPUs fail loudly instead of silently falling back to scalar behavior.
- **D-06:** Generation-stamped handles remain mandatory for safe close/reuse semantics.

### Parser lifecycle and reuse
- **D-07:** `parser_parse` must hard-fail with `ERR_PARSER_BUSY` when the parser already owns a live `Doc`. Re-parse does not implicitly invalidate the prior `Doc`.
- **D-08:** `Doc.Close()` is the explicit release point that clears the parser-busy state. Old parser/doc handles still rely on generation checks to turn stale use into a clean error rather than UB.

### Error and diagnostics contract
- **D-09:** The primary ABI contract is stable `int32` error codes on every export. No struct-by-value returns and no stringly-typed primary failure path.
- **D-10:** The ABI should include a small diagnostics surface in addition to numeric codes: ABI version handshake, active implementation name, and bounded diagnostic helpers for parse context such as last-error text and byte offset.
- **D-11:** Diagnostic helpers are advisory only. Control flow must remain driven by numeric error codes so bindings stay portable across all five targets.

### Handle and value model
- **D-12:** `Parser` and `Doc` remain opaque generation-stamped `u64` handles.
- **D-13:** `Element`, `Array`, and `Object` are lightweight views tied to a `Doc` plus native node/index state. They are not independently allocated/freeable opaque handles.
- **D-14:** Value-view results cross the ABI through out-params / iterator state, not struct-by-value returns.

### Read-side ABI shape
- **D-15:** String access in `v0.1` is copy-out only. The contract should use explicit `ptr + len` results with a matching free path rather than borrowed zero-copy string views.
- **D-16:** Array and object traversal use stateful iterator/cursor entry points driven from Go. `next()` returns key/value or node references through out-params.
- **D-17:** Direct field lookup remains part of the ABI via explicit lookup helpers such as `object_get_field`, returning value-view state through out-params rather than materializing Go-native structures.

### Claude's Discretion
- Exact numeric allocation of the error-code ranges, as long as the space remains stable and documented.
- Final naming of helper functions and iterator structs, as long as they preserve the decisions above.
- Whether bounded diagnostics use thread-local scratch space or explicit copy-out buffers, as long as they remain advisory rather than primary control flow.

### Deferred Ideas (OUT OF SCOPE)
- Zero-copy string views tied to `Doc` lifetime remain a `v0.2` concern, not part of the `v0.1` FFI contract.
- Richer diagnostics and introspection beyond a minimal helper surface should be revisited only if post-MVP demand justifies the added ABI weight.

## Summary

Phase 1 is not just prose. Because `FFI-01` requires a committed, `cbindgen`-generated header, the plan needs a minimal Rust ABI source of truth in this phase even if functional parsing stays deferred. The repo currently has no `Cargo.toml`, no `go.mod`, no test harness, and no existing source tree; `cbindgen` is also not installed locally. [CITED: https://github.com/mozilla/cbindgen] [VERIFIED: local file audit via `find . -maxdepth 4 -type f`] [VERIFIED: local tool audit]

The stable contract should stay intentionally boring: every export returns `int32_t`; all data comes back through out-params; no struct-by-value returns; no mixed float and integer argument lists; `Parser` and `Doc` are the only opaque handles; `Element`/`Array`/`Object` stay lightweight views; `parser_parse` fails with `ERR_PARSER_BUSY` while a live `Doc` exists; and every parse copies the input into Rust-owned padded memory. That shape matches the locked project decisions, purego’s current ABI limits, and simdjson DOM’s parser/document lifetime model. [VERIFIED: .planning/phases/01-ffi-contract-design/01-CONTEXT.md] [CITED: https://github.com/ebitengine/purego] [CITED: https://raw.githubusercontent.com/ebitengine/purego/main/func.go] [CITED: https://simdjson.org/api/1.0.0/md_doc_dom.html] [CITED: https://simdjson.org/api/0.9.0/classsimdjson_1_1dom_1_1parser.html]

The contract should also document two implementation obligations now, even though enforcement lands in later phases: every Rust export uses an `ffi_fn!`-style panic boundary, and every C++ seam uses non-throwing simdjson access (`.get(err)`-style) so no foreign exception crosses into Rust. The planner should phrase these as policy plus future verification steps, not as a promise that release-mode `panic=abort` will become recoverable error codes. [CITED: https://doc.rust-lang.org/nomicon/ffi.html] [CITED: https://simdjson.org/api/1.0.0/md_doc_dom.html] [VERIFIED: .planning/REQUIREMENTS.md]

**Primary recommendation:** Plan Phase 1 as an ABI-spec phase with three deliverables created together: a minimal ABI-source Rust crate, `include/pure_simdjson.h`, and `docs/ffi-contract.md`, all verified by the same static contract checks. [CITED: https://github.com/mozilla/cbindgen] [VERIFIED: .planning/ROADMAP.md]

### Pattern 1: Lowest-Common-Denominator Export Signatures

**What:** Every exported function should return `int32_t` and use pointer out-params for results; no struct-by-value returns; no mixed float/integer argument lists. [CITED: https://github.com/ebitengine/purego] [CITED: https://raw.githubusercontent.com/ebitengine/purego/main/func.go]

**When to use:** All public ABI exports in `include/pure_simdjson.h`. [VERIFIED: .planning/REQUIREMENTS.md]

**Example:**
```c
// Sources: purego v0.10.0 platform/support docs and func.go ABI logic.
typedef uint64_t pure_simdjson_handle_t;

typedef struct pure_simdjson_value_view {
  pure_simdjson_handle_t doc;
  uint64_t state0;
  uint64_t state1;
  uint32_t kind;
  uint32_t reserved;
} pure_simdjson_value_view_t;

int32_t pure_simdjson_parser_new(pure_simdjson_handle_t *out_parser);
int32_t pure_simdjson_parser_parse(
    pure_simdjson_handle_t parser,
    const uint8_t *input_ptr,
    size_t input_len,
    pure_simdjson_handle_t *out_doc);
int32_t pure_simdjson_doc_root(
    pure_simdjson_handle_t doc,
    pure_simdjson_value_view_t *out_root);
```

### Pattern 2: Parser/Doc as Handles, Values as Views

**What:** Keep `Parser` and `Doc` as generation-stamped opaque `u64` handles, but model `Element`/`Array`/`Object` as small copyable view structs tied to a `Doc`, never as independently freeable handles. [VERIFIED: .planning/phases/01-ffi-contract-design/01-CONTEXT.md] [VERIFIED: .planning/research/ARCHITECTURE.md]

**When to use:** Any value produced from `doc_root`, iterators, or `object_get_field`. [VERIFIED: .planning/phases/01-ffi-contract-design/01-CONTEXT.md]

**Example:**
```c
// Sources: locked D-12..D-14 decisions; layout below is recommended, not yet locked.
typedef uint64_t pure_simdjson_handle_t; /* slot:u32 | gen:u32 */

typedef struct pure_simdjson_object_iter {
  pure_simdjson_handle_t doc;
  uint64_t cursor0;
  uint64_t cursor1;
} pure_simdjson_object_iter_t;

int32_t pure_simdjson_object_iter_new(
    const pure_simdjson_value_view_t *object_view,
    pure_simdjson_object_iter_t *out_iter);
```

### Pattern 3: Explicit Busy-State Parser Lifecycle

**What:** Treat parser reuse as a state machine: one parser owns at most one live document; `parser_parse` returns `ERR_PARSER_BUSY` until `doc_close` succeeds; generation checks still reject stale handles after close. This is stricter than raw simdjson reuse and prevents silent document replacement across the FFI boundary. [CITED: https://simdjson.org/api/0.9.0/classsimdjson_1_1dom_1_1parser.html] [VERIFIED: .planning/phases/01-ffi-contract-design/01-CONTEXT.md]

**When to use:** `parser_parse`, `doc_close`, and every API that accepts a `Doc` or view. [VERIFIED: .planning/phases/01-ffi-contract-design/01-CONTEXT.md]

**Example:**
```text
parser_new -> parser_parse -> doc_root -> ... -> doc_close -> parser_parse
parser_new -> parser_parse -> parser_parse == ERR_PARSER_BUSY
```

### Pattern 4: Copy-In, Copy-Out Data Crossing

**What:** Keep borrowed memory out of the public `v0.1` ABI. Parse copies into Rust-owned padded memory; string and diagnostic helpers should use caller-owned copy-out buffers so Go never relies on borrowed native pointers with hidden lifetime rules. The copy-in half is locked; the copy-out helper form is still a design recommendation. [VERIFIED: .planning/phases/01-ffi-contract-design/01-CONTEXT.md] [CITED: https://simdjson.org/api/0.9.0/classsimdjson_1_1dom_1_1parser.html] [ASSUMED]

**When to use:** `parser_parse`, `element_get_string`, `get_last_error_text`, and any future diagnostic helpers. [VERIFIED: .planning/phases/01-ffi-contract-design/01-CONTEXT.md] [ASSUMED]

**Example:**
```c
// Sources: simdjson parser padding rules; copy-out buffer shape is recommended.
int32_t pure_simdjson_element_get_string_size(
    const pure_simdjson_value_view_t *view,
    size_t *out_len);

int32_t pure_simdjson_element_copy_string(
    const pure_simdjson_value_view_t *view,
    uint8_t *dst,
    size_t dst_cap,
    size_t *out_written);
```

### Anti-Patterns to Avoid

- **Docs-only contract phase:** `cbindgen` needs a Rust crate with a public C API, so prose alone cannot satisfy `FFI-01`. [CITED: https://github.com/mozilla/cbindgen]  
- **Raw pointer handles exposed to Go:** stale reuse and double-free become memory safety bugs instead of deterministic contract errors. [VERIFIED: .planning/phases/01-ffi-contract-design/01-CONTEXT.md] [VERIFIED: .planning/research/PITFALLS.md]  
- **Borrowed string views in `v0.1`:** they make lifetime bugs easy and directly violate locked scope. [VERIFIED: .planning/phases/01-ffi-contract-design/01-CONTEXT.md]  
- **Promise that release panics become error codes:** Rust explicitly says `panic=abort` still aborts immediately. [CITED: https://doc.rust-lang.org/nomicon/ffi.html]  
- **Native-to-Go callback iteration:** locked out already, and it conflicts with the project’s FFI risk research. [VERIFIED: .planning/phases/01-ffi-contract-design/01-CONTEXT.md] [VERIFIED: .planning/research/PITFALLS.md]  

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Header synchronization | Hand-edited `include/pure_simdjson.h` | `cbindgen` + committed generated header + diff check | The generator is designed specifically to reflect Rust C-ABI items and reduce drift. [CITED: https://github.com/mozilla/cbindgen] |
| Compound-value transport | Struct returns or “clever” platform-specific calling conventions | `int32_t` return + out-param structs | purego’s ABI support is still platform-dependent; lowest-common-denominator signatures avoid cross-target surprises. [CITED: https://github.com/ebitengine/purego] [CITED: https://raw.githubusercontent.com/ebitengine/purego/main/func.go] |
| Resource identity | Raw pointers reused as stable public handles | Generation-stamped packed `u64` handles | This is already the project’s locked safety model and cleanly turns stale use into `ERR_INVALID_HANDLE`. [VERIFIED: .planning/phases/01-ffi-contract-design/01-CONTEXT.md] |
| Diagnostics lifetime | Global borrowed `char *` buffers | Bounded caller-owned copy-out helpers | Copy-out buffers are the most portable and testable option for purego-style bindings. [ASSUMED] |
| Iteration bridge | Native callbacks into Go | Go-driven cursor/iterator state | It keeps control flow in Go and matches locked scope. [VERIFIED: .planning/phases/01-ffi-contract-design/01-CONTEXT.md] |

**Key insight:** The Phase 1 contract should optimize for portability and auditability, not minimum call count. Higher-level ergonomics belong in the Go wrapper, not the raw ABI. [VERIFIED: .planning/PROJECT.md] [ASSUMED]

## Common Pitfalls

### Pitfall 1: Treating Phase 1 as “write docs first, code later”
**What goes wrong:** The phase finishes with prose but no machine-generated header source of truth, so `FFI-01` cannot actually be diff-checked. [CITED: https://github.com/mozilla/cbindgen]  
**Why it happens:** `cbindgen` operates on a Rust crate with a public C API, not on Markdown. [CITED: https://github.com/mozilla/cbindgen]  
**How to avoid:** Reserve Wave 0 for a minimal ABI-only Rust crate and `cbindgen.toml` before finalizing signatures. [CITED: https://github.com/mozilla/cbindgen] [ASSUMED]  
**Warning signs:** The phase has `docs/ffi-contract.md` but no `Cargo.toml`, `src/lib.rs`, or generated `include/pure_simdjson.h`. [VERIFIED: local file audit via `find . -maxdepth 4 -type f`]  

### Pitfall 2: Assuming a `Doc` survives a second parse on the same parser
**What goes wrong:** Old value views silently point at overwritten parser-owned state. [CITED: https://simdjson.org/api/0.9.0/classsimdjson_1_1dom_1_1parser.html]  
**Why it happens:** simdjson DOM explicitly reuses one parser’s internal buffers for a single live document. [CITED: https://simdjson.org/api/1.0.0/md_doc_dom.html]  
**How to avoid:** Keep the project’s locked `ERR_PARSER_BUSY` contract and make `Doc.Close()` the only release point. [VERIFIED: .planning/phases/01-ffi-contract-design/01-CONTEXT.md]  
**Warning signs:** Any proposed signature or call sequence allows `parser_parse` while a `Doc` handle from that parser is still valid. [VERIFIED: .planning/phases/01-ffi-contract-design/01-CONTEXT.md]  

### Pitfall 3: Overstating `catch_unwind`
**What goes wrong:** The contract claims all Rust panics become numeric error codes even in release builds. [CITED: https://doc.rust-lang.org/nomicon/ffi.html]  
**Why it happens:** `catch_unwind` catches unwinding panics, but `panic=abort` still aborts immediately. [CITED: https://doc.rust-lang.org/nomicon/ffi.html]  
**How to avoid:** Document `ffi_fn!` as mandatory for unwind-mode builds and debugging, while separately documenting `panic=abort` as the release policy. [CITED: https://doc.rust-lang.org/nomicon/ffi.html] [VERIFIED: .planning/REQUIREMENTS.md]  
**Warning signs:** Contract text says “panic returns `ERR_PANIC` in release” without qualification. [CITED: https://doc.rust-lang.org/nomicon/ffi.html]  

### Pitfall 4: Assuming purego’s newer struct support means richer ABIs are safe everywhere
**What goes wrong:** Signatures work on Linux/macOS and then misbehave on Windows or fallback code paths. [CITED: https://github.com/ebitengine/purego/releases/tag/v0.10.0] [CITED: https://raw.githubusercontent.com/ebitengine/purego/main/func.go]  
**Why it happens:** `v0.10.0` added Linux struct support, but purego still documents platform tiers and a Windows fallback that may not support floats. [CITED: https://github.com/ebitengine/purego] [CITED: https://raw.githubusercontent.com/ebitengine/purego/main/func.go]  
**How to avoid:** Keep the ABI on the safest subset: `int32_t` returns, out-params, no mixed float/int arg lists. [CITED: https://raw.githubusercontent.com/ebitengine/purego/main/func.go]  
**Warning signs:** A proposed export returns a struct, or takes both `double` and integer arguments in the same signature. [CITED: https://raw.githubusercontent.com/ebitengine/purego/main/func.go]  

### Pitfall 5: Leaving exception or borrowed-buffer policy implicit
**What goes wrong:** C++ exceptions or borrowed buffer/string lifetimes leak across the seam and later phases guess differently. [CITED: https://simdjson.org/api/1.0.0/md_doc_dom.html] [CITED: https://doc.rust-lang.org/nomicon/ffi.html]  
**Why it happens:** simdjson offers both exception and non-exception styles, and DOM parsing can copy or rely on padded input depending on entry point. [CITED: https://simdjson.org/api/1.0.0/md_doc_dom.html] [CITED: https://simdjson.org/api/0.9.0/classsimdjson_1_1dom_1_1parser.html]  
**How to avoid:** State two non-negotiables in the contract: `.get(err)`/non-throwing use only, and Rust-owned padded input on every parse. [VERIFIED: .planning/phases/01-ffi-contract-design/01-CONTEXT.md] [CITED: https://simdjson.org/api/1.0.0/md_doc_dom.html]  
**Warning signs:** Contract text says “implementation-defined,” “borrowed string,” or “exception may propagate.” [VERIFIED: .planning/phases/01-ffi-contract-design/01-CONTEXT.md]

## Open Questions

1. **How should the public numeric error-code space be allocated?**
   What we know: the ABI must expose stable `int32` error codes, while simdjson already defines parser/accessor errors such as `UNSUPPORTED_ARCHITECTURE`, `NUMBER_OUT_OF_RANGE`, `NO_SUCH_FIELD`, `PARSER_IN_USE`, and `INSUFFICIENT_PADDING`. [CITED: https://simdjson.org/api/3.0.0/namespacesimdjson.html]
   What's unclear: whether public ABI numbers should mirror upstream simdjson numeric values or reserve project-owned ranges and translate upstream internally. [ASSUMED]
   Recommendation: reserve project-owned public ranges and document the mapping table in `docs/ffi-contract.md`; do not let upstream numeric values become the stable public ABI by accident. [ASSUMED]

2. **Should diagnostics use explicit copy-out buffers or thread-local scratch storage?**
   What we know: the diagnostics surface is locked as advisory-only, and the discretion area explicitly leaves this implementation choice open. [VERIFIED: .planning/phases/01-ffi-contract-design/01-CONTEXT.md]
   What's unclear: whether TLS scratch simplifies call sites enough to justify hidden lifetime rules and potential concurrency/debugging complications. [ASSUMED]
   Recommendation: lock explicit copy-out buffers now unless there is a measured ergonomics problem. [ASSUMED]

3. **What is the smallest stable view-state layout that still supports root, lookup, and iteration helpers?**
   What we know: `Element`/`Array`/`Object` must be lightweight views, not independently freeable handles. [VERIFIED: .planning/phases/01-ffi-contract-design/01-CONTEXT.md]
   What's unclear: whether one shared `value_view` struct is enough, or whether iterators and object/array views need distinct repr(C) structs for clarity and future-proofing. [ASSUMED]
   Recommendation: decide this before writing `cbindgen.toml`, because this choice affects header readability, linter complexity, and Go mirror types. [ASSUMED]

### Known Threat Patterns for this stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Stale or double-freed handle reused after close | Tampering | Generation-stamped packed `u64` handles and idempotent close semantics. [VERIFIED: .planning/phases/01-ffi-contract-design/01-CONTEXT.md] |
| Rust panic or C++ exception escaping across FFI | Denial of Service | `ffi_fn!` / `catch_unwind` policy for unwinding builds, and non-throwing simdjson seam only. [CITED: https://doc.rust-lang.org/nomicon/ffi.html] [CITED: https://simdjson.org/api/1.0.0/md_doc_dom.html] |
| Unowned or unpadded input buffer crossing parse boundary | Tampering | Rust-owned padded copy on every parse. [VERIFIED: .planning/phases/01-ffi-contract-design/01-CONTEXT.md] [CITED: https://simdjson.org/api/0.9.0/classsimdjson_1_1dom_1_1parser.html] |
| DLL search-order hijack when later loading Windows artifacts | Elevation of Privilege | Full-path loading and `LoadLibraryEx`-style safe directory flags in later loader phases. [CITED: https://learn.microsoft.com/en-us/windows/win32/dlls/dynamic-link-library-search-order] [CITED: https://learn.microsoft.com/en-us/windows/win32/api/libloaderapi/nf-libloaderapi-loadlibraryexa] |
| Oversized copy-out into undersized caller buffer | Tampering | Two-step “size query, then copy” helpers or required/written length out-params. [ASSUMED] |
