## M1 — activeLibrary() double-checked locking (locked design)

The existing `activeLibrary()` acquires `libraryMu` before calling
`resolveLibraryPath()`. That is fine today because resolve is in-process, but after
this phase resolve can trigger a multi-minute bootstrap download. Holding the
process-wide loader mutex across network I/O serializes every concurrent
`NewParser()` call behind the first one's bandwidth — unacceptable.

Rewrite pattern (standard double-checked locking):

```go
func activeLibrary() (*loadedLibrary, error) {
	// Fast path: read cached pointer under the lock, release immediately.
	libraryMu.Lock()
	cached := cachedLibrary
	libraryMu.Unlock()
	if cached != nil {
		return cached, nil
	}

	// Slow path: resolve (may download) WITHOUT holding libraryMu.
	// M1: network I/O never happens under the loader-wide mutex.
	path, attempted, err := resolveLibraryPath()
	if err != nil {
		return nil, wrapLoadFailure("resolve library path", fmt.Errorf("%w; tried: %s",
			err, formatAttemptedPaths(attempted)))
	}
	handle, err := loadLibrary(path)
	if err != nil {
		return nil, wrapLoadFailure("dlopen", err)
	}
	bindings, implName, err := ffi.Bind(handle, func(name string) (uintptr, error) {
		return lookupSymbol(handle, name)
	})
	if err != nil {
		return nil, wrapLoadFailure("bind symbols", err)
	}

	// Re-check under the lock and install. If a concurrent caller beat us, drop
	// our handle and return theirs. The extra dlopen in a race is cheap; the
	// alternative is holding the lock across the whole download window (M1).
	libraryMu.Lock()
	defer libraryMu.Unlock()
	if cachedLibrary != nil {
		// A concurrent caller won. Discard our handle.
		// TODO(v0.2): if purego exposes dlclose, call it here. For v0.1 leaking
		// on a race is acceptable because it happens at most once per process.
		return cachedLibrary, nil
	}
	cachedLibrary = &loadedLibrary{
		path:               path,
		handle:             handle,
		implementationName: implName,
		bindings:           bindings,
	}
	return cachedLibrary, nil
}
```

Key invariants this pattern preserves:
- `libraryMu` is held only for the read at line 1 and the recheck+insert block
  (tens of nanoseconds).
- `resolveLibraryPath()`, `loadLibrary()`, `ffi.Bind()` all run outside the mutex.
- Result pointer identity is stable after first successful load (cachedLibrary
  never reassigned once non-nil).
- Double-init races are benign: the losing caller discards its handle; only the
  first installation takes effect.

**M1 acceptance rule (grep-verifiable):** In the rewritten `activeLibrary()`, the
line containing `resolveLibraryPath()` MUST appear OUTSIDE any `libraryMu.Lock()`
section. The test below asserts this.

        absPath, err := filepath.Abs(envPath)

    cachePath := bootstrap.CachePath(runtime.GOOS, runtime.GOARCH)
    if _, err := os.Stat(cachePath); err == nil {
        return cachePath, []string{cachePath}, nil

    // Stage 3: auto-bootstrap
    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Minute)
    defer cancel()

    }
    // Stage 4: cache hit after bootstrap
    if _, err := os.Stat(cachePath); err == nil {
